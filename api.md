# API contract — лендинг ↔ backend ↔ бот

Контракт связи между фронтендом (Next.js на Vercel) и backend (Supabase Postgres + сервисные роуты + Telegram-бот).

Источник решения: `docs/backend-decision.md`. Архитектура: `docs/architecture.md`. Правила проекта: `.claude/CLAUDE.md`.

Документ описывает **намеренный контракт**, не текущую реализацию. По мере появления кода каждый эндпоинт получает ссылку на файл реализации.

---

## Общие правила

- **Транспорт:** HTTPS, JSON.
- **Кодировка:** UTF-8.
- **Часовой пояс:** все timestamps — `UTC ISO 8601` (например, `2026-05-18T12:34:56Z`).
- **Идентификаторы:** UUID v4 (`lead_id`, `user_id`, `broadcast_id`).
- **Версионирование:** префикс `/api/` без версии в URL. Брейкинг-изменения — через новый путь.
- **Ошибки:** единый формат (см. ниже).
- **Аутентификация лендинга → backend:** простой `Content-Type: application/json` + защита от спама (rate limit + honeypot). Cookies/JWT в MVP не используются.
- **Аутентификация бота → backend:** не нужна, бот общается с БД напрямую через `DATABASE_URL`.
- **Аутентификация webhook backend → бот:** заголовок `X-Bot-Webhook-Secret: <BOT_WEBHOOK_SECRET>`.

## Формат ошибок

Все эндпоинты возвращают ошибки в одном формате:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Поле email обязательно",
    "details": {
      "field": "email"
    }
  }
}
```

| Код                  | HTTP | Когда                                                          |
|----------------------|------|----------------------------------------------------------------|
| `VALIDATION_ERROR`   | 400  | Запрос не прошёл валидацию схемы                               |
| `RATE_LIMITED`       | 429  | Слишком много запросов с одного IP / на один эндпоинт          |
| `UNAUTHORIZED`       | 401  | Нет валидного секрета на защищённом эндпоинте                  |
| `NOT_FOUND`          | 404  | Ресурс не найден                                               |
| `CONFLICT`           | 409  | Конфликт состояния (например, повторная заявка)                |
| `INTERNAL_ERROR`     | 500  | Непредвиденная ошибка backend                                  |

`error.message` — человекочитаемый русский текст, безопасный для показа пользователю.
`error.details` — опциональный объект для отладки, может отсутствовать.

## Эндпоинты

### POST `/api/leads` — приём заявки с лендинга

Создаёт запись в таблице `leads` и инициирует уведомление админу в Telegram-бот.

**Кто вызывает:** форма заявки на лендинге (`web/`).
**Куда пишет:** Supabase Postgres → таблица `leads`.
**Побочный эффект:** отправка уведомления админу через бот (через `BOT_WEBHOOK` или прямой вызов Telegram Bot API из роута — решается на этапе 3).

#### Запрос

```http
POST /api/leads
Content-Type: application/json
```

```json
{
  "name": "Александр",
  "contact": "@username | +7900... | email",
  "message": "Хочу диагностику",
  "source": "landing-hero",
  "honeypot": ""
}
```

Поля:

| Поле        | Тип      | Обязательное | Правило                                                        |
|-------------|----------|--------------|----------------------------------------------------------------|
| `name`      | string   | да           | 2..80 символов, trim                                           |
| `contact`   | string   | да           | 3..120 символов, любой формат (telegram / phone / email)       |
| `message`   | string   | нет          | 0..2000 символов                                               |
| `source`    | string   | нет          | идентификатор секции/CTA (`landing-hero`, `landing-cta`, ...)  |
| `honeypot`  | string   | да           | должен быть пустым; если заполнен — заявка молча отбрасывается |

#### Ответ — 200 OK

```json
{
  "ok": true,
  "lead_id": "8a3c0e6e-1f3e-4a3b-9d8b-2a8b7e9d0c11",
  "created_at": "2026-05-18T12:34:56Z"
}
```

#### Ошибки

- `400 VALIDATION_ERROR` — невалидные поля.
- `429 RATE_LIMITED` — превышен лимит (например, > 5 заявок / минуту с IP).
- `500 INTERNAL_ERROR` — БД недоступна или сбой уведомления (заявка всё равно сохраняется, ошибка только в нотификации логируется отдельно).

### POST `/api/bot/notify` — внутренний webhook backend → бот *(опционально)*

Используется, если решим вынести уведомление админу из роута `/api/leads` в отдельный шаг (например, retry-логика).

**Кто вызывает:** Next.js Route Handler `/api/leads`.
**Куда:** Python-бот (FastAPI-эндпоинт рядом с aiogram) или сам Telegram Bot API.

#### Запрос

```http
POST /api/bot/notify
Content-Type: application/json
X-Bot-Webhook-Secret: <BOT_WEBHOOK_SECRET>
```

```json
{
  "event": "lead.created",
  "lead_id": "8a3c0e6e-1f3e-4a3b-9d8b-2a8b7e9d0c11"
}
```

#### Ответ

```json
{ "ok": true }
```

Если решение по доставке уведомлений будет «звать Telegram Bot API напрямую из Next.js», этот эндпоинт отменяется — фиксируем тогда отдельно.

## Что НЕ входит в контракт MVP

- Эндпоинты для просмотра/изменения заявок — админ работает с заявками **внутри Telegram-бота** через прямое подключение к Supabase. Отдельного REST-API для админки нет.
- Auth-флоу для админа — whitelist `ADMIN_IDS` на стороне бота.
- Платежи, подписки, личный кабинет — за рамками MVP.

## Связь с таблицами БД (предварительно)

Контракт ссылается на сущности, которые будут описаны в `db/schema.md` на этапе 5:

- `leads` — поля минимум: `id`, `name`, `contact`, `message`, `source`, `status`, `created_at`, `updated_at`.
- `users` — пользователи бота (`telegram_id`, `username`, `is_admin`, ...).
- `broadcasts`, `audit_log` — появятся на этапах 3–4.

## Изменения контракта

Любое изменение этого файла — это изменение публичного контракта между лендингом и backend. Правило:

1. Описать изменение в PR-комментарии.
2. Обновить `api.md` в том же PR.
3. Если изменение брейкинговое — упомянуть это в `tasks/lessons.md`.
