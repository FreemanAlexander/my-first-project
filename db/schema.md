# Database schema — MVP

Схема базы данных для учебного проекта AgentOS (продающий лендинг + Telegram-бот с админкой).

Источники: `docs/backend-decision.md` (БД — Supabase Postgres), `api.md` (контракт лендинг ↔ backend), `docs/bot-flows.md` (сценарии бота), `docs/architecture.md` (модули).

Документ описывает **намерение**, не миграции. SQL-миграции и подключение к Supabase появятся на отдельном шаге этапа 5 в `tasks/todo.md`. Сейчас ключи и проект Supabase не заводим.

---

## Соглашения

- **Имена:** `snake_case`. Таблицы во множественном числе, кроме лог-таблиц (`audit_log`).
- **Идентификаторы заявок:** `UUID v4` (`id uuid`). Для лога — `bigserial`. Для админов — `telegram_id` (целое от Telegram).
- **Время:** все колонки времени — `timestamptz`, значения — в UTC. Суффикс `_at` (`created_at`, `updated_at`, `accepted_at`, `done_at`, `notified_at`).
- **Булевы:** префикс `is_` (`is_active`).
- **Перечисления:** не используем `enum` Postgres — храним как `text` + `CHECK`-констрейнт. Легче эволюционировать без миграций типа.
- **JSON:** только `jsonb`. Применяем точечно, не для замены нормализации.
- **Источник правды для авторизации админа:** env-переменная `ADMIN_IDS` в боте. Таблица `admins` — это мирор для отображаемых данных и FK из `audit_log`, не источник прав.

---

## Обзор сущностей

```text
admins  ──┐
          │
          ▼
       audit_log
          ▲
          │
          └── leads
```

- `leads` — заявки с лендинга. Главный поток данных.
- `audit_log` — журнал действий администраторов (по заявкам и не только).
- `admins` — справочник админов для отображения и FK из `audit_log`.

В MVP **не заводим**:
- `users` (пользователи бота-гости),
- `broadcasts` и `broadcast_recipients` (рассылки),
- `faq_items` (FAQ хранится в коде бота).

Причины — в разделе «Что откладываем».

---

## Таблица `leads`

Заявки с лендинга. Создаётся по `POST /api/leads` (см. `api.md`). Обрабатывается админом в боте.

| Колонка        | Тип                   | NULL | Default       | Назначение                                     |
|----------------|-----------------------|------|---------------|------------------------------------------------|
| `id`           | `uuid`                | no   | `gen_random_uuid()` | первичный ключ заявки                    |
| `name`         | `text`                | no   | —             | имя из формы, **PII**                          |
| `contact`      | `text`                | no   | —             | контакт (tg/phone/email), **PII**              |
| `message`      | `text`                | yes  | —             | сообщение из формы, **PII** (может содержать)  |
| `source`       | `text`                | yes  | —             | идентификатор секции/CTA (`landing-hero`, ...) |
| `status`       | `text`                | no   | `'new'`       | состояние заявки (см. ниже)                    |
| `created_at`   | `timestamptz`         | no   | `now()`       | момент создания                                |
| `updated_at`   | `timestamptz`         | no   | `now()`       | обновляется триггером при любом UPDATE         |
| `notified_at`  | `timestamptz`         | yes  | —             | момент успешной доставки уведомления админу    |
| `accepted_at`  | `timestamptz`         | yes  | —             | первый переход в `in_progress`                 |
| `done_at`      | `timestamptz`         | yes  | —             | момент закрытия (`done` или `rejected`)        |
| `accepted_by`  | `bigint`              | yes  | —             | FK → `admins.telegram_id`                      |
| `done_by`      | `bigint`              | yes  | —             | FK → `admins.telegram_id`                      |

**Констрейнты:**

- `CHECK (char_length(name) BETWEEN 2 AND 80)`
- `CHECK (char_length(contact) BETWEEN 3 AND 120)`
- `CHECK (message IS NULL OR char_length(message) <= 2000)`
- `CHECK (status IN ('new', 'in_progress', 'done', 'rejected'))`
- `FOREIGN KEY (accepted_by) REFERENCES admins(telegram_id) ON DELETE SET NULL`
- `FOREIGN KEY (done_by) REFERENCES admins(telegram_id) ON DELETE SET NULL`

**Не валидируем `contact` регуляркой.** Он может быть `@username`, телефоном, e-mail или произвольной строкой. Разбор формата — на стороне бота при показе кнопок «Связаться с клиентом» (см. `docs/bot-flows.md`).

## Таблица `audit_log`

Журнал действий админа: что, по какой заявке, когда. Это **не** журнал всех событий системы — только намеренные действия с правами.

| Колонка               | Тип            | NULL | Default        | Назначение                                       |
|-----------------------|----------------|------|----------------|--------------------------------------------------|
| `id`                  | `bigserial`    | no   | autoincrement  | PK                                               |
| `admin_telegram_id`   | `bigint`       | no   | —              | кто (FK → `admins.telegram_id`)                  |
| `action`              | `text`         | no   | —              | код действия (`lead.accept`, `lead.done`, ...)   |
| `lead_id`             | `uuid`         | yes  | —              | FK → `leads.id`, NULL для действий не по заявке  |
| `payload`             | `jsonb`        | yes  | —              | детали действия (комментарий, прежнее значение)  |
| `created_at`          | `timestamptz`  | no   | `now()`        | момент действия                                  |

**Констрейнты:**

- `FOREIGN KEY (admin_telegram_id) REFERENCES admins(telegram_id) ON DELETE RESTRICT`
- `FOREIGN KEY (lead_id) REFERENCES leads(id) ON DELETE SET NULL`
  *(удаление заявки не должно молча выпиливать историю — оставляем запись с `lead_id = NULL` и `payload`-снимком)*
- `CHECK (action ~ '^[a-z_]+\.[a-z_]+$')` — действия в формате `domain.event`.

**Допустимые значения `action` в MVP:**

| Action                  | Смысл                                                                |
|-------------------------|----------------------------------------------------------------------|
| `lead.accept`           | админ нажал «Принять в работу»                                       |
| `lead.done`             | админ нажал «Отметить обработанной»                                  |
| `lead.reject`           | админ отклонил заявку                                                |
| `lead.contact_opened`   | админ нажал «Связаться с клиентом» (факт перехода, не отправка)      |

Новые `action`-коды добавляются сюда вместе с обновлением `docs/bot-flows.md`.

**Что в `payload`:** короткий контекст события, не дублирующий поля заявки. Примеры:

```json
// lead.done с комментарием
{ "comment": "созвонились, перевели на курс" }

// lead.accept без комментария
{}

// lead.contact_opened с типом канала
{ "channel": "telegram" }
```

**В `payload` НЕ кладём:** контактные данные клиента, текст исходного сообщения, токены, секреты, полный объект заявки. Это дублирование и риск утечки PII через лог.

## Таблица `admins`

Справочник админов. Источник прав — env `ADMIN_IDS`, эта таблица — мирор для отображения имени и FK из `audit_log`.

| Колонка        | Тип            | NULL | Default       | Назначение                                  |
|----------------|----------------|------|---------------|---------------------------------------------|
| `telegram_id`  | `bigint`       | no   | —             | PK, Telegram user id                        |
| `username`     | `text`         | yes  | —             | `@username` без `@`, опционально            |
| `display_name` | `text`         | yes  | —             | имя для отображения в карточках             |
| `is_active`    | `boolean`      | no   | `true`        | мягкое отключение без удаления записи       |
| `created_at`   | `timestamptz`  | no   | `now()`       | момент добавления                           |

**Заполнение:**

- На старте бота для каждого `telegram_id` из `ADMIN_IDS` делаем `INSERT ... ON CONFLICT DO NOTHING`.
- `username` / `display_name` обновляются при первом действии админа из апдейта Telegram.
- **Удаление админа = снятие `is_active`**, не `DELETE`. Это сохраняет ссылочную целостность `audit_log`.

**Авторизация не идёт через эту таблицу.** Каждое нажатие админской кнопки проверяет `from_user.id ∈ ADMIN_IDS` (env), а не `SELECT FROM admins`. Это решение зафиксировано в `docs/bot-flows.md` и не меняется этой схемой.

## Статусы заявки

```text
new ──► in_progress ──► done
  │             │
  │             └────► rejected
  └─────────────────► rejected
```

| Статус         | Что значит                                                | Кто переводит сюда           |
|----------------|-----------------------------------------------------------|------------------------------|
| `new`          | Заявка только что пришла, никто не взялся                 | система (по `INSERT`)        |
| `in_progress`  | Админ принял в работу                                     | админ (`[Принять]`)          |
| `done`         | Завершена успешно                                         | админ (`[Обработано]`)       |
| `rejected`     | Отклонена (спам, дубль, не по теме)                       | админ (`[Отклонить]`)        |

**Правила переходов:**

- `new → in_progress`: устанавливается `accepted_at`, `accepted_by`.
- `new → done`, `in_progress → done`: устанавливается `done_at`, `done_by`.
- `new → rejected`, `in_progress → rejected`: устанавливается `done_at`, `done_by`.
- Откат `done → in_progress` или `done → new` в MVP **не разрешаем**. Если ошибся — отдельная новая заявка или ручная правка через SQL с записью в `audit_log` действия `lead.reopen` *(добавится при необходимости)*.

Все переходы пишутся в `audit_log` соответствующим `action`.

## PII — где она

Под PII в этом проекте понимаем данные, по которым можно идентифицировать или связаться с конкретным человеком.

| Колонка                  | PII? | Комментарий                                               |
|--------------------------|------|-----------------------------------------------------------|
| `leads.name`             | да   | имя из формы                                              |
| `leads.contact`          | да   | контакт клиента (tg/phone/email)                          |
| `leads.message`          | да*  | свободный текст, может содержать любые данные             |
| `leads.source`           | нет  | технический идентификатор секции                          |
| `leads.status`           | нет  | состояние                                                 |
| `leads.*_at` / `*_by`    | нет  | временные метки и id админов                              |
| `admins.username`        | да   | telegram-handle админа                                    |
| `admins.display_name`    | да   | имя админа                                                |
| `admins.telegram_id`     | да   | идентификатор человека в Telegram                         |
| `audit_log.payload`      | да*  | может содержать комментарий с PII — см. правило ниже      |

**Правила обращения с PII:**

- В логи приложения PII **не пишем** в виде plaintext. ID заявки и `audit_log.id` — можно.
- Доступ к таблицам — только через `SUPABASE_SERVICE_ROLE_KEY` на сервере (Next.js Route Handlers, Python-бот). В браузер сервисный ключ не выходит. Это уже зафиксировано в `docs/backend-decision.md`.
- В `callback_data` Telegram — только `lead_id` (UUID), никаких `name` / `contact`.
- Удаление по запросу клиента: ставим заглушки в `name`, `contact`, `message` (например, `__deleted__`), сохраняем `id` и `status` для целостности `audit_log`. Полный `DELETE` не используем.

## Индексы

Только те, которые понадобятся в реальных запросах MVP. Без преждевременной оптимизации.

**`leads`:**

- PK `(id)` — авто.
- `CREATE INDEX leads_status_created_at_idx ON leads (status, created_at DESC);`
  *Покрывает «список заявок с фильтром по статусу и сортировкой по дате», основной запрос админ-меню.*
- `CREATE INDEX leads_created_at_idx ON leads (created_at DESC);`
  *Для «всех заявок» без фильтра.*

**`audit_log`:**

- PK `(id)` — авто.
- `CREATE INDEX audit_log_lead_id_created_at_idx ON audit_log (lead_id, created_at DESC);`
  *История по конкретной заявке (карточка заявки в боте).*
- `CREATE INDEX audit_log_admin_created_at_idx ON audit_log (admin_telegram_id, created_at DESC);`
  *Активность конкретного админа (отчёты, аудит).*

**`admins`:**

- PK `(telegram_id)` — авто.
- Дополнительных индексов нет: таблица маленькая, ходим по PK.

**Что НЕ добавляем сейчас:**

- Индексы на `name`, `contact`, `message` — поиск по PII в MVP не делаем.
- Полнотекстовый поиск — пока не нужен.
- Частичные индексы вроде `WHERE status = 'new'` — добавим, если профилирование покажет.

## Что хранить нельзя

- **Секреты приложения:** `TELEGRAM_BOT_TOKEN`, `DATABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `BOT_WEBHOOK_SECRET`. Живут только в env, никогда не в таблицах.
- **Платёжные данные:** номера карт, CVV, данные счетов. В MVP нет платежей.
- **Государственные идентификаторы:** паспортные данные, СНИЛС, ИНН и подобное. Форма их не собирает.
- **Хешированные пароли:** в проекте нет паролей.
- **Полная переписка с клиентом:** мы храним только то, что клиент сам прислал через форму (`message`). Личная переписка админа с клиентом ведётся в Telegram/телефоне/почте и в БД не зеркалится.
- **Биометрия, медицинская и финансовая чувствительная информация:** проект для этого не предназначен.
- **Свободные комментарии с чужой PII в `audit_log.payload`:** ответственность админа держать комментарии короткими и без данных третьих лиц.

Если требование «хранить X» появится — заводим отдельную таблицу с обоснованием и обновляем этот раздел.

## Что откладываем на будущее

- **`users` — пользователи бота вне админов.** В MVP БД таблицу не заводим: `/start` в первой итерации stateless для гостя, что согласовано с `docs/bot-flows.md`. Таблица появится вместе с рассылками или сегментацией — тогда оба документа обновляем синхронно (включая `users.opt_out` для `/stop`).
- **`broadcasts` + `broadcast_recipients`.** Рассылки в MVP — заглушка в боте (см. `docs/bot-flows.md`).
- **`faq_items`.** FAQ хранится в коде бота, миграция в БД — после стабилизации списка вопросов.
- **`opt_out` / `consent` поля.** Понадобятся вместе с рассылками. Сейчас формы не запрашивают подписку на рассылки.
- **RLS (Row Level Security) политики.** Доступ к БД идёт только через сервисный ключ на сервере — RLS можно подключить, но в MVP она не критична. Включаем при появлении клиентского SDK-доступа.
- **Партиционирование `audit_log`.** При росте лога более чем на десятки тысяч строк в месяц — партиционируем по `created_at`. На MVP объёмах это преждевременно.
- **Soft-delete заявок (`deleted_at`).** Сейчас удаление через зануление PII (см. раздел про PII). Если появится requirement на «корзину» — добавим колонку.
- **Триггер `updated_at`.** На уровне Postgres обычно делается через триггер `BEFORE UPDATE`. Создаётся в первой миграции этапа 5, в этом документе только фиксируем намерение.
