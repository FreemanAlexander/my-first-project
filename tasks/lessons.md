## 2026-05-19 — sanitize-check падает в upstream public-gbrain-agentos (осознанное отклонение)

**Что не так.**
Запустили `bash scripts/sanitize-check.sh` в `~/gbrain-readonly` (Step 0 из `docs/setup.md`).
Скрипт упал с exit 1: нашёл «запрещённые» кодовые имена агентов (`silvana`, `kaelthas`, `thrall`, `illidan`, `tyrande`) в:
- `tests/test_hmac_review_fixes.py`, `tests/test_recall_mcp.py`, `tests/test_hmac_auth.py`, `tests/test_swarm_worker_hmac.py`, `tests/test_gbrain_doctor.py`, `tests/test_hmac_format_parity.py`
- `docs/INTER-AGENT-WEBHOOKS.md`
- `DEVIATIONS.md`
- `.env.example`

В `services/` (боевой код MCP-сервисов) совпадений нет.

Гайд требует: «If sanitize-check exits 0, the repo is clean. If not, re-clone and try again. Do not edit by hand to make it pass.» (`docs/setup.md:22`).

**Диагностика (read-only).**
- `git remote -v` → `origin = https://github.com/qwwiwi/public-gbrain-agentos.git` (правильный публичный репо).
- `git branch --show-current` → `main`.
- `git log --oneline -5` → `HEAD = 6d252ab (grafted, HEAD -> main, origin/main, origin/HEAD)`.
- `git status --short` → пусто, локальных правок нет.
- Клон shallow (`grafted`), но содержимое = upstream `main`.

Вывод: это **не наш битый клон**, а рассинхронизация в самом upstream public repo: автор не прогнал свой же publish-gate перед коммитом `6d252ab`, либо забыл выкосить кодовые имена из tests/docs.

**Как правильно по букве guide.**
Re-clone не поможет (мы и так на `origin/main`). Самый чистый путь — issue в `qwwiwi/public-gbrain-agentos` и ждать фикс. Руками править файлы public-репо или sanitize-check **запрещено** гайдом и нашими правилами (`.claude/CLAUDE.md` → Security: ничего не патчим в чужих репах под себя).

**Что решили (осознанное отклонение).**
Продолжаем установку, **не правя ни одного файла** в `~/gbrain-readonly` и не модифицируя `sanitize-check.sh`. Основание: все совпадения в тестах/документации/примерах, сервисный код чист, функциональный риск отсутствует. Отклонение зафиксировано здесь.

**Правило на будущее.**
1. Если документированный preflight-гейт падает, но HEAD == origin и локальных правок нет — проблема в upstream, не в клоне. Re-clone в этом случае бессмыслен.
2. Любой отказ от шага из guide фиксируется в `tasks/lessons.md` с коммитом/источником, до того как двинуться дальше.
3. Чужие репозитории (особенно read-only зеркала вроде `~/gbrain-readonly`) **не правим**, даже чтобы прогнать их же проверку. Лучше задокументированное отклонение, чем тихий патч.

## 2026-05-19 — Уроки установки gbrain на gbrain-vps

Контекст: первая установка gbrain из upstream public repo на VPS. Ниже — что всплыло по ходу, что починили и какие правила выносим на будущее.

### 1. sanitize-check (upstream, коммит 6d252ab)

**Что не так.** `bash scripts/sanitize-check.sh` падает в `~/gbrain-readonly` на коммите `6d252ab`. Совпадения только в `tests/`, `docs/`, `DEVIATIONS.md`, `.env.example`. В `services/` (сервисный код MCP) чисто.

**Как правильно.** Не патчим upstream и не модифицируем sanitize-check. Продолжаем install осознанно — функциональный риск отсутствует, отклонение задокументировано (см. блок выше).

**Правило.** Если preflight-гейт upstream падает, но сервисный код чист — фиксируем отклонение и идём дальше, не правя чужой репо.

### 2. Засвет admin-token при install

**Что не так.** В ходе install в чат/лог попал старый admin-token: фильтр секретов не поймал base64url-формат токена (символы `-`/`_` вместо `+`/`/`).

**Как правильно.** Старый token сразу отозвали, выпустили новый. Новое значение лежит **только** в `/opt/gbrain/secrets/admin.token` и в моём менеджере паролей. В чат, stdout и логи токен не выводится. Команды вида `cat /opt/gbrain/secrets/admin.token` — запрещены.

**Правило.** Регэксп секрет-фильтра обязан покрывать `base64url` (`[A-Za-z0-9_-]{N,}`), не только классический base64. Любой засвеченный токен — немедленный rotate.

### 3. Баг install: отсутствует /opt/gbrain/vault

**Что не так.** После `install-vps.sh` сервисы gbrain упали: каталог `/opt/gbrain/vault` не был создан скриптом, хотя сервисы ожидают его на старте.

**Как правильно (минимальный фикс).**
1. Создать `/opt/gbrain/vault`.
2. Скопировать содержимое из `vault-template` (рядом в `/opt/gbrain/`).
3. Выставить владельца: `chown -R gbrain:gbrain /opt/gbrain/vault`.
4. Перезапустить сервисы gbrain.

**Правило.** Если service unit падает с ошибкой по пути — сначала проверяем, что путь существует и принадлежит сервисному пользователю, прежде чем лезть в код или конфиги. И не запускаем `install-vps.sh` повторно поверх рабочей системы — баг чиним точечно.

### 4. smoke-test.sh устарел относительно MCP Streamable HTTP

**Что не так.** `smoke-test.sh` из upstream рапортует FAIL на `/mcp`: ответ HTTP 400 `Missing session ID`. При этом сами сервисы живые и порты слушают.

**Как правильно.** Это не падение сервиса, а устаревший smoke: новый MCP Streamable HTTP требует session handshake перед запросом, а скрипт его не делает. Health сервисов проверяем иначе (статусы systemd, порты, ручной handshake), на красный smoke не реагируем как на инцидент.

**Правило.** Smoke-скрипты upstream — не источник истины. Если smoke красный, а сервисы и порты живые — смотрим, не отстал ли smoke от протокола.

### 5. MCP-порты были открыты на 0.0.0.0 → закрыли через UFW

**Что не так.** Сервисы MCP (`8766`, `8767`, `8768`) изначально слушали на `0.0.0.0`, то есть на всех интерфейсах включая публичный.

**Как правильно.** Включили UFW со следующими правилами:
- `allow 22/tcp` — SSH оставить доступным;
- `deny 8766/tcp`, `deny 8767/tcp`, `deny 8768/tcp` — MCP-порты закрыть снаружи.

Доступ к MCP — только через локальный loopback / туннель, не публично.

**Правило.** Любой новый сервис, который слушает на `0.0.0.0`, по умолчанию режем фаерволом, пока не доказана необходимость публичного доступа. Bind на `127.0.0.1` — предпочтительнее, чем `0.0.0.0` + UFW deny, но если bind не настраиваем сейчас — хотя бы deny на фаерволе.

### 6. Swap 4 GB на gbrain-vps

**Что не так.** VPS без swap уязвим к OOM-kill, особенно при индексации / прогреве MCP.

**Как правильно.** Создали swap-файл 4 GB как страховку. Не подменяет нормальный мониторинг памяти, но снимает риск внезапного kill сервисов под пиковую нагрузку.

**Правило.** На VPS без swap — добавлять swap до первого тяжёлого workload, не после первого OOM.

### 7. Admin-token хранится только вне репозитория

**Что не так.** Соблазн положить admin-token «временно» в conversation/файл/коммит для удобства.

**Как правильно.** Admin-token сохранён вручную в менеджер паролей. На сервере он лежит в `/opt/gbrain/secrets/admin.token` с правами `gbrain:gbrain`. В чат значение не выводится. В git-репозиторий — никогда.

**Правило.** Секреты живут в `secrets/` на сервере и в менеджере паролей. Любой запрос «покажи токен» / `cat .../admin.token` — отклоняем.
