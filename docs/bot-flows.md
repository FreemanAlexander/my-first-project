# Telegram bot flows — MVP

Сценарии Telegram-бота для учебного проекта AgentOS (продающий лендинг + бот с админкой).

Источники: `.claude/CLAUDE.md`, `docs/architecture.md`, `docs/backend-decision.md`, `api.md`.

Документ описывает **поведение**, не код. Стек: Python 3.12 + aiogram 3.x, БД — Supabase Postgres, доступ через `DATABASE_URL`. Реализация — отдельные пункты в `tasks/todo.md` (раздел «3. Backend / Telegram-бот»).

---

## Роли пользователей бота

| Роль       | Кто это                              | Как распознаём                       |
|------------|--------------------------------------|--------------------------------------|
| `guest`    | Любой человек, написавший боту       | По умолчанию                         |
| `admin`    | Владелец/оператор проекта            | `telegram_id` в whitelist `ADMIN_IDS`|

Whitelist `ADMIN_IDS` живёт в `.env` бота, в репозиторий не попадает. Любой callback с админскими действиями проверяет `from_user.id ∈ ADMIN_IDS` **на стороне бота, при каждом нажатии**. Без серверной проверки кнопок «принять / обработано» нельзя — Telegram-клиент не доверенная среда.

## Сценарий: `/start`

**Триггер:** пользователь открывает чат с ботом или отправляет команду `/start`.

**Поведение бота:**

1. Определить роль: `admin`, если `telegram_id ∈ ADMIN_IDS`, иначе `guest`. Источник прав — только env-whitelist (см. `db/schema.md`, раздел про `admins`).
2. Отправить приветственное сообщение:
   - короткий текст в стиле `design.md` (без капса, без восклицаний);
   - бренд-знак ПМЛ как часть приветствия (текстовый или приложенным изображением — решается на этапе вёрстки бота).
3. Показать reply-меню по роли:
   - **guest:** `[Узнать о проекте]` `[Оставить заявку]` `[FAQ]`
   - **admin:** `[Заявки]` `[FAQ]` `[Рассылки]` *(пункт «Рассылки» — заглушка «появится позже»)*.

**`/start` в MVP — stateless для гостя.** Таблицы `users` в MVP БД нет (см. `db/schema.md`, раздел «Что откладываем»), поэтому upsert по `telegram_id` не делаем: запоминать гостя пока негде и незачем. Бот реагирует на каждый `/start` заново. Для админа состояние всё равно мирорится в `admins` при первом действии, а не на `/start`.

**Глубокие ссылки `?start=<payload>`:**

- `?start=lead-<lead_id>` — открыть карточку конкретной заявки. Доступно только админу; гостю отвечаем «недостаточно прав», без подробностей.
- `?start=faq-<slug>` — открыть конкретный пункт FAQ.

**Что НЕ делаем на `/start`:**

- Не показываем «весь функционал сразу» — короткое приветствие + меню.
- Не запрашиваем телефон/контакты автоматически.
- Не пишем в `audit_log` — это обычное действие пользователя, не админа.

## Сценарий: заявка с лендинга → уведомление админу

**Триггер:** успешный `POST /api/leads` на стороне Next.js. Контракт описан в `api.md`.

**Путь данных:**

```text
Лендинг (форма)
  → POST /api/leads (Next.js Route Handler)
    → INSERT INTO leads (Supabase Postgres)
      → уведомление в Telegram-бот
        → сообщение каждому admin из ADMIN_IDS
```

**Способ доставки уведомления — на выбор (фиксируется в этапе 3):**

- **Вариант A.** Из Next.js напрямую вызываем Telegram Bot API (`sendMessage`) для каждого `admin_id`. Бот в этой ветке только отвечает на нажатия кнопок.
- **Вариант B.** Next.js шлёт `POST /api/bot/notify` с `X-Bot-Webhook-Secret` (см. `api.md`), бот сам формирует сообщение. Удобно, если нужны retry/очередь и единая точка форматирования.

В MVP стартуем с варианта A как с самого простого. Если появится очередь/retry — переключаемся на B без изменения контракта.

**Содержимое сообщения админу:**

```
Новая заявка #<short_id>
Имя: <name>
Контакт: <contact>          ← в исходном виде, без маскирования (это приватный чат админа)
Источник: <source>
Сообщение:
<message или прочерк, если пусто>

Создано: <created_at в часовом поясе админа>
```

Inline-клавиатура под сообщением:

- `[Принять в работу]`  → `lead:accept:<lead_id>`
- `[Отметить обработанной]` → `lead:done:<lead_id>`
- `[Открыть карточку]` → `lead:open:<lead_id>`

`callback_data` — короткий и стабильный, формат `<scope>:<action>:<lead_id>`. Никаких PII в `callback_data` (Telegram хранит её до 64 байт и она видна в апдейтах).

**Идемпотентность.** Если по одному `lead_id` уведомление уже было доставлено (например, ретрай со стороны бэка) — повторно не шлём. Признак отправки храним в таблице `leads` (поле `notified_at`) или в отдельной `lead_notifications`. Финальная форма — на этапе 5.

**Сбой уведомления.** Заявка считается принятой, даже если уведомление не доставлено. Ошибка нотификации логируется отдельно, заявка не теряется — её всегда видно в списке `[Заявки]`.

## Сценарий: просмотр заявки админом

**Точки входа:**

1. Нажатие `[Открыть карточку]` под уведомлением.
2. Команда/кнопка `[Заявки]` → список → выбор заявки.
3. Deep link `t.me/<bot>?start=lead-<lead_id>`.

**Карточка заявки содержит:**

- `id` (короткий, читаемый), `status` (`new` / `in_progress` / `done` / `rejected`),
- `name`, `contact`, `source`,
- полный `message`,
- `created_at`, `updated_at`,
- метаданные согласия на ОПД: `consent_policy_version` и `consent_given_at` одной строкой (например, «Согласие на ОПД: v2026-05-18, 12:34 UTC»). `consent_ip` / `consent_user_agent` в карточке **не показываем** — это техданные, доступ к ним только через БД,
- последние 5 записей из `audit_log` по этой заявке (кто, что, когда).

**Inline-кнопки на карточке:**

- `[Принять в работу]` — активна при `status = new`.
- `[Отметить обработанной]` — активна при `status ∈ { new, in_progress }`.
- `[Связаться с клиентом]` — всегда активна, см. сценарий ниже.
- `[Назад к списку]` — возврат в список заявок.

Неактивные кнопки либо скрываются, либо показываются «бледно» (Telegram не поддерживает «disabled» нативно — реализуем через скрытие).

**Список `[Заявки]`:**

- Сортировка по `created_at DESC`.
- Фильтры (кнопки сверху): `Все` / `Новые` / `В работе` / `Обработаны`.
- Пагинация: 10 заявок на страницу, кнопки `← Назад` / `Вперёд →`.

## Сценарий: базовые админские действия

Все действия:

1. Проверяют `from_user.id ∈ ADMIN_IDS`. Если нет — `answer_callback_query("недостаточно прав")`, ничего больше.
2. Делают `UPDATE leads SET ...`.
3. Пишут запись в `audit_log` (поля минимум: `admin_id`, `action`, `lead_id`, `payload`, `created_at`).
4. Обновляют сообщение карточки (`edit_message_text` / `edit_message_reply_markup`) — пользователь сразу видит новый статус.
5. Отвечают `answer_callback_query` с короткой подтверждающей подписью (например, «принято в работу»).

### Принять в работу

- Условие: `status = new`. Иначе — короткое сообщение «уже не «новая»».
- Эффект: `status` → `in_progress`, `updated_at = now()`, `audit_log.action = "lead.accept"`.

### Отметить обработанной

- Условие: `status ∈ { new, in_progress }`.
- Эффект: `status` → `done`, `updated_at = now()`, `audit_log.action = "lead.done"`.
- Допустимо запросить короткий комментарий (FSM на 1 шаг: «коротко опишите итог, /skip — пропустить»). Комментарий уходит в `audit_log.payload`.

### Написать клиенту вручную

В MVP бот **не пишет клиенту от себя**. Это сознательное решение и с точки зрения 152-ФЗ: согласие, зафиксированное в `leads.consent_personal_data`, покрывает обработку заявки (связь с клиентом по оставленному им контакту), но не даёт права на автоматическую рассылку или массовый outreach. Контакт мог быть телефоном/почтой, а не Telegram — автоответ от бота сюда тоже не подходит.

Поведение `[Связаться с клиентом]`:

- Если `contact` похож на `@username` или числовой `telegram_id` → формируем кнопку с deep link `tg://resolve?domain=<username>` или `tg://user?id=<id>`. Админ нажимает и попадает в личный чат.
- Если `contact` похож на телефон → показываем номер моноширинно (для удобства копирования) и кнопку `tel:`-ссылки.
- Если `contact` похож на email → моноширинный текст + кнопка `mailto:`-ссылки.
- Если формат неоднозначный → просто показываем `contact` моноширинно, без кнопок.

После нажатия кнопки пишем в `audit_log` событие `lead.contact_opened` — это не отправка сообщения клиенту, а факт перехода админа в личный канал связи.

## Сценарий: FAQ

**Триггеры:** команда `/faq`, кнопка `[FAQ]` в меню, deep link `?start=faq-<slug>`.

**Поведение:**

1. Показать список вопросов из FAQ-блока лендинга (источник — `docs/landing-sections.md`, секция FAQ).
2. Каждый вопрос — отдельная inline-кнопка с `callback_data = faq:show:<slug>`.
3. По клику бот шлёт текст ответа + кнопку `[← К списку FAQ]`.

**Где хранятся вопросы:**

- В MVP — в коде бота (статический словарь), чтобы не плодить сущности. Поменялся FAQ на лендинге → синхронно меняем в боте.
- На следующем шаге, если FAQ растёт, переезжаем в таблицу `faq_items` в Supabase (за рамками MVP).

**Безопасное поведение:** FAQ ничего не пишет в БД, не требует прав, доступен всем.

## Сценарий: рассылки *(будущий этап, не реализуем сейчас)*

Описано здесь, чтобы зафиксировать намерение и не путать в текущей итерации.

**Что планируется (этап после MVP):**

- Только админ. Гость пункт меню не видит.
- Сегменты: `все пользователи бота`, `только с заявками`, `только обработанные`.
- Флоу: черновик → предпросмотр → подтверждение → отправка пачками с rate-limit.
- Аудит: каждая рассылка пишется в `broadcasts` + соответствующие записи в `audit_log`.
- Compliance: вместе с рассылками появится таблица `users` с полем `opt_out`; команда `/stop` отписывает пользователя, рассылки не идут тем, у кого `opt_out = true`. В MVP БД таблицы `users` ещё нет (см. `db/schema.md`).

**Сейчас:** пункт меню «Рассылки» показываем как заглушку с текстом «появится позже». Никакой реальной отправки.

## Ошибки и безопасное поведение

### Авторизация

- Любой админский callback проверяет `from_user.id ∈ ADMIN_IDS`. Это происходит **в каждом обработчике**, не только при входе в меню.
- Whitelist живёт в env-переменной бота. В код константой не вшиваем.
- Никакой авторизации через «секретное слово» в чате — только Telegram-ID.

### Секреты и PII

- Токен бота, `DATABASE_URL`, `INTERNAL_BOT_NOTIFY_SECRET` — только в `.env`. В логи, сообщения и `callback_data` не попадают **никогда**.
- В логах PII (телефон, email, имя, `consent_ip`, `consent_user_agent`) — либо маскируем (`+7***1234`), либо не пишем. ID заявки в логах — можно.
- В групповые чаты и публичные каналы бот сообщения о заявках не шлёт — только в личку конкретному админу.

### Согласие на ОПД (152-ФЗ)

- Бот считает, что **любая** заявка в `leads` уже прошла проверку согласия на стороне Next.js. Без `consent_personal_data = true` запись в БД не появляется (см. `api.md`, `CONSENT_REQUIRED`).
- Бот **не отправляет** клиенту сообщений от своего имени и не использует `contact` для маркетинговых сценариев. Объём согласия — только обработка конкретной заявки, не рассылки.
- При удалении заявки по запросу клиента (см. `db/schema.md`) — `consent_*` метаданные (версия, момент) сохраняются как доказательство, а PII зануляется. Бот показывает такую заявку как анонимизированную, без `name`/`contact`.

### Webhook от backend → бот

- Если используем `POST /api/bot/notify`, проверяем заголовок `X-Bot-Webhook-Secret` на полное совпадение, **до** парсинга тела.
- Невалидный секрет → `401 UNAUTHORIZED`, в логи пишем только факт (без значений секрета и без тела запроса).

### Сбои

| Сбой                              | Как ведёт себя бот                                          |
|-----------------------------------|-------------------------------------------------------------|
| БД недоступна                     | «временно недоступно, попробуйте позже», error → логи       |
| Telegram API временно недоступен  | Retry с экспоненциальным backoff (1s, 2s, 4s, до 3 попыток)|
| Неожиданное исключение в handler  | `answer_callback_query("ошибка, мы уже знаем")`, traceback в логи (без секретов) |
| Невалидный `callback_data`        | Игнорируем (`answer_callback_query()` без текста), warning в логи |
| Спам в `/start`                   | Простой rate-limit: не чаще 1 раза в 3 секунды на `telegram_id` |

### Идемпотентность кнопок

- Двойное нажатие `[Принять в работу]` не должно приводить к двум записям в `audit_log`. Проверяем текущий `status` перед `UPDATE` и пишем audit только при реальной смене состояния.
- При повторном нажатии — короткое сообщение «уже принято».

### Логирование

- Структурные логи (JSON-строки) с полями: `level`, `timestamp`, `event`, `lead_id`, `admin_id`, `action`. Без `token`, без `contact`, без `message_text`.
- Уровни: `info` для обычных действий, `warning` для ожидаемых отказов (нет прав, невалидный callback), `error` для исключений и сбоев БД/API.
