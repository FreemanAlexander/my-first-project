# Backend decision — MVP

Документ фиксирует выбор backend-платформы для учебного проекта AgentOS (продающий лендинг + Telegram-бот с админкой).

Источник правил: `.claude/CLAUDE.md`. Связанные документы: `docs/architecture.md`, `api.md`, `db/schema.md` (появится позже).

## Решение

Решение двухступенчатое — sandbox и production живут на разных платформах. Граница — наличие **реальных персональных данных граждан РФ**.

### Sandbox / MVP / learning — Supabase

- Хранилище данных: Supabase Postgres (хостинг вне РФ, AWS-регионы).
- Серверная логика лендинга: Next.js Route Handlers (`web/app/api/*`) на Vercel.
- Бизнес-логика бота: Python/aiogram, подключается к тому же Supabase Postgres.
- Доступ к БД с лендинга — только через серверные роуты Next.js с `SUPABASE_SERVICE_ROLE_KEY`. Сервисный ключ в браузер не выходит.
- **Жёсткое ограничение:** на Supabase **нельзя** заводить реальные заявки с настоящих ПДн граждан РФ. Только синтетические данные, dev-аккаунты, демо-формы. Это учебный sandbox, не прод.

### Production (ПМЛ с реальными заявками) — Yandex Cloud

- Хранилище данных: **Managed Service for PostgreSQL** в Yandex Cloud (или другой RU-hosted Postgres). Серверы в РФ.
- Серверная логика лендинга: остаётся на Next.js / Vercel, но `DATABASE_URL` ведёт в РФ-инстанс.
- Бот: тот же Python/aiogram, `DATABASE_URL` тот же РФ-инстанс.
- Подключение — по TLS, credentials в secrets хостинга, не в репозитории.
- **Почему так:** 152-ФЗ требует, чтобы первичная запись, систематизация и хранение ПДн граждан РФ велись в БД, физически расположенной в РФ. Подробнее — раздел «152-ФЗ и локализация ПДн» ниже.

Переход с Supabase на Yandex Cloud — это **смена `DATABASE_URL` и хостинга**, не смена кода: схема Postgres одинаковая, миграции одни и те же. Решение сознательное — чтобы не платить за переписывание при выходе в прод.

## Почему Supabase для sandbox

1. **Минимум DevOps на старте.** Managed Postgres, не нужно поднимать сервер, настраивать бэкапы и сеть. На учебной фазе ценно.
2. **Совместимость с Vercel.** Vercel + Supabase — стандартная пара под Next.js, есть готовая интеграция и документация.
3. **Учебная ценность.** Будут затронуты: SQL, миграции, RLS (Row Level Security), сервисные ключи vs anon ключи.
4. **Тот же диалект Postgres, что и в Yandex Cloud.** Схема и миграции не переписываются при переходе в прод — меняется только `DATABASE_URL` и место хостинга.
5. **Доступ из Python-бота.** Обычный Postgres с TLS-подключением. aiogram-бот подключится через `asyncpg` / SQLAlchemy.

## Почему Yandex Cloud для production

1. **Локализация ПДн (152-ФЗ).** Первичная запись и хранение ПДн граждан РФ должны вестись на серверах в РФ. Managed PostgreSQL в Yandex Cloud это закрывает «из коробки».
2. **Тот же Postgres.** Миграции и схема из sandbox едут как есть. Не нужно переписывать ORM, запросы, типы.
3. **Backup / monitoring / restore.** Managed-сервис снимает с нас бэкапы и мониторинг кластера — это критично, когда в БД настоящие заявки клиентов.
4. **РФ-юрисдикция и SLA.** Договор и SLA с резидентом РФ, что упрощает работу с подрядчиками и согласия.

Альтернативы Yandex Cloud (VK Cloud, SberCloud, Cloud.ru, собственный VPS с PostgreSQL в РФ) допустимы — критерий один: **сервер физически в РФ**. Если будет выбрана не Yandex Cloud, обновляем этот раздел и `docs/architecture.md`.

## Почему не другие варианты для prod

### Firebase — не подходит

- Firestore — документная NoSQL-база. Реляции `lead → user → broadcast → audit_log` придётся эмулировать через дублирование данных и cloud functions.
- Auth и Realtime — сильные стороны Firebase, но в нашем MVP они не критичны: админ логинится через Telegram, форма заявки не требует realtime.
- Когда оправдано: realtime-чат, мобильное приложение с offline-режимом, простая схема без сильных связей.

### VPS + SQLite — слишком много ручной работы

- Плюсы: полный контроль, минимальная стоимость, простая модель.
- Минусы:
  - SQLite ограничен по конкурентным записям (бот + лендинг = два процесса пишут в БД).
  - Бэкапы, миграции, restart-логика, TLS — всё руками.
  - На VPS отдельно нужно держать сервис, мониторинг, healthcheck.
- Когда оправдано: однопроцессные инструменты, локальные утилиты, прототип без сети.

## 152-ФЗ и локализация ПДн

Архитектурный compliance-комплект, который фиксируется этим решением. **Полный юридический пакет** (уведомление РКН, регламент инцидентов, ответственный за ПДн, точные сроки хранения по целям, локальные акты) **в эту итерацию не входит** — он списком висит в `tasks/todo.md` как production-checklist.

### Что считаем ПДн в этом проекте

Имя, контакт (telegram/phone/email) и свободный `message` из формы заявки — это ПДн. Подробная карта PII — в `db/schema.md`.

### Локализация (152-ФЗ ст. 18 ч. 5)

- **Production:** первичная запись, систематизация и хранение ПДн — в РФ. Реализуем через Yandex Cloud Managed PostgreSQL (или другой RU-hosted Postgres).
- **Sandbox (Supabase):** реальные ПДн **запрещены**. Учебный sandbox принимает только синтетические данные. На лендинг-форме в окружениях `local` и `preview` это можно подкрепить баннером «тестовая среда».
- **Трансграничная передача** в sandbox формально не идёт — там нет реальных субъектов.

### Согласие на обработку ПДн

- Лендинг-форма в production **обязана** иметь чекбокс согласия и ссылку на Privacy Policy / согласие на ОПД. Поле `consent_personal_data` — обязательное в запросе, без него заявка не принимается.
- На стороне сервера фиксируем версию согласия и метаданные подписи (момент времени, IP, User-Agent) — это закрепляется в `db/schema.md` колонками `consent_*` таблицы `leads` и в `api.md` соответствующими полями.
- Изменение текста согласия = новая версия `consent_policy_version`. Старые согласия не «обновляются задним числом».

### Чего НЕ делаем в этой итерации

- Не пишем полный текст Privacy Policy / согласия — это юридический документ, его создаём отдельно перед прод-релизом.
- Не подаём уведомление в РКН и не назначаем ответственного за ПДн — это шаг pre-prod checklist.
- Не реализуем процедуру запросов субъекта (доступ/удаление/блокировка) — добавим, когда понадобится. До тех пор ручная процедура через контакт админа.

## Что фиксируется этим решением

- Двухступенчатая платформа: **sandbox = Supabase (без реальных ПДн)**, **prod = Yandex Cloud Managed PostgreSQL (с реальными ПДн)**.
- Схема Postgres одна и та же для обеих сред — миграции переезжают без переписывания.
- Сервисные ключи и `DATABASE_URL` живут только на сервере (Next.js Route Handlers и Python-бот). В браузер не попадают.
- Доступ из лендинга к БД — **через Next.js API**, не через клиентский Supabase SDK. Это даёт единый контракт `api.md` и единое место для проверки `consent_personal_data`.
- Доступ бота к БД — напрямую через `DATABASE_URL`, без HTTP-слоя.
- `consent_*` поля в `leads` — часть архитектурного контракта на обоих этапах (см. `db/schema.md`, `api.md`).

## Что НЕ фиксируется этим решением

- Конкретный ORM/DB-layer (Drizzle vs Prisma на web, SQLAlchemy vs asyncpg в боте) — выбор отложен до этапа 5 (Database) в `tasks/todo.md`.
- Конкретный RU-провайдер для prod — пока «Yandex Cloud Managed PostgreSQL» по умолчанию. VK Cloud / SberCloud / Cloud.ru / собственный РФ-VPS — допустимы, если будут веские причины. При смене провайдера обновляется этот файл и `docs/architecture.md`.
- Точные сроки хранения по каждой цели обработки, регламент инцидентов, шаблоны согласий, текст Privacy Policy — это юридический пакет, не входит в эту итерацию (см. production checklist в `tasks/todo.md`).
- Edge Functions / RLS-политики — добавляются по мере необходимости, когда появится конкретный сценарий.
- Платформа для хостинга бота (VPS / Fly.io в РФ) — выбор отложен до этапа 7 (Deploy).
- Аутентификация админа в боте — реализуется через whitelist `ADMIN_IDS`, отдельный Supabase Auth не используется в MVP.

## Переменные окружения (план)

Реальные значения в репозиторий не попадают. В `.env.example` отражаются только имена ключей.

**Sandbox / MVP:**

```text
# Web (Next.js, на Vercel)
SUPABASE_URL=
SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=    # только серверные роуты, никогда в браузер

# Bot (Python/aiogram)
TELEGRAM_BOT_TOKEN=
DATABASE_URL=                  # Postgres connection string из Supabase
ADMIN_IDS=                     # whitelist через запятую

# Связь web ↔ bot (если понадобится webhook со стороны лендинга в бот)
WEB_BASE_URL=
INTERNAL_BOT_NOTIFY_SECRET=    # секрет для X-Bot-Webhook-Secret header

# 152-ФЗ
CONSENT_POLICY_VERSION=        # версия согласия на ОПД (например, "2026-05-18")
```

**Production добавляет / заменяет:**

```text
# DB переезжает в РФ
DATABASE_URL=                  # connection string к Yandex Cloud Managed PostgreSQL
DATABASE_SSL_ROOT_CERT_PATH=   # путь к корневому сертификату RU-провайдера

# Переменные SUPABASE_* в prod не используются
```

Сейчас секреты НЕ заводим: ни проект Supabase, ни Yandex Cloud не создаются на этом шаге.
