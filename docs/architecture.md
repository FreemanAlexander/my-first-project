# Architecture — обзор проекта

Короткое описание модулей учебного проекта AgentOS: продающий лендинг + Telegram-бот с админскими функциями. Источник визуальных правил — `design.md`, правила Claude Code — `.claude/CLAUDE.md`, план задач — `tasks/todo.md`.

## Модули

### `web/` — лендинг
- Продающий лендинг на **Next.js 15 (App Router)** + **TypeScript 5.x (strict)**.
- Отвечает за публичную часть: hero, манифест, методика, тарифы, FAQ, форма заявки/CTA.
- Использует design tokens из `design.md` (цвета, типографика, spacing).
- Не хранит состояние пользователей: при отправке формы передаёт заявку дальше — в бот/backend и/или напрямую в Supabase.
- Деплой: **Vercel**.

### `bot/` — Telegram-бот
- Бот на **Python 3.12** + **aiogram 3.x**.
- Основной runtime бизнес-логики MVP: приём заявок с лендинга, уведомления админу, FAQ, рассылки.
- Здесь же живёт **админка MVP**: команды и inline-кнопки для администратора (см. ниже).
- При необходимости рядом поднимаем **FastAPI**-сервис для приёма webhook от лендинга и/или внешних интеграций.
- Логи — структурные, без токенов и PII.

### `db/` — данные
- **Supabase Postgres** как единое хранилище для лендинга и бота.
- В каталоге `db/` лежат схема (`db/schema.md`), миграции и при необходимости seed-данные.
- Основные сущности (планово): `users`, `leads`, `broadcasts`, `audit_log`.
- Доступ к БД — через сервисные ключи на стороне бота/backend, не из браузера.

### `docs/` — документация
- Сквозная документация по проекту: архитектура, секции лендинга, сценарии бота, схема БД, локальный запуск, тесты, деплой.
- Этот файл (`docs/architecture.md`) — точка входа для нового участника проекта.

## Пользовательский путь

```text
Посетитель → Лендинг (web/) → Форма / CTA
        → Заявка уходит в backend (bot/ или FastAPI рядом)
        → Запись в Supabase (db/, таблица leads)
        → Уведомление админу в Telegram-бот (bot/)
        → Админ обрабатывает заявку командами/кнопками в боте
```

Ключевое: лендинг — это «витрина», бот — это «оператор и админка», Supabase — единое хранилище состояния.

## Где живёт админка

На MVP отдельной веб-админки **нет**. Все админские функции — внутри Telegram-бота:
- список и карточки заявок,
- действия по заявке: Принять / Отложить / Отклонить,
- рассылки,
- аудит-лог действий админа,
- whitelist по `admin_id`.

Отдельный каталог `admin/` пока не создаём. Если позже понадобится отдельный фронт админки, решение фиксируем в `docs/admin-decision.md` и только тогда добавляем модуль.

## Отложенные решения

Эти выборы намеренно отложены, чтобы не фиксировать стек слишком рано:

- **ORM / DB layer.** Кандидаты для web — Drizzle или Prisma; для Python — SQLAlchemy или asyncpg. Финальный выбор — перед этапом 5 (Database) из `tasks/todo.md`.
- **Деплой бота / backend.** Кандидаты — VPS или Fly.io. Финальный выбор — перед этапом 7 (Deploy) из `tasks/todo.md`. Лендинг по умолчанию — Vercel.
